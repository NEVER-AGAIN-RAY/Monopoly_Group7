# Monopoly Deal — 功能需求与非功能需求文档

> 本文档依据《Monopoly Deal》（地产大亨卡牌版）官方规则书整理，列出系统开发所需的全部功能需求（Functional Requirements）与非功能需求（Non-functional Requirements），作为后续项目开发的基础依据。

---

## 一、游戏基本规则概述

《Monopoly Deal》是一款 2‑5 人卡牌游戏。目标：**率先收集到 3 套完整的地产颜色套装**即获胜。
牌组共 110 张，分为：金钱牌、地产牌、行动牌（含租金牌）。

---

## 二、功能需求（Functional Requirements）

### 2.1 玩家管理

| 编号 | 描述 |
|------|------|
| FR-01 | 系统支持 2 至 5 名玩家参与同一局游戏。 |
| FR-02 | 每位玩家拥有独立的手牌区、银行区（金钱堆）和地产区。 |
| FR-03 | 游戏开始时，系统向每位玩家随机发放 5 张手牌。 |
| FR-04 | 系统维护并展示当前回合玩家的身份（如姓名或编号）。 |
| FR-05 | 系统支持设置玩家姓名或昵称。 |

### 2.2 牌组管理

| 编号 | 描述 |
|------|------|
| FR-06 | 系统维护一副标准的 110 张 Monopoly Deal 牌组，并在游戏开始时随机洗牌。 |
| FR-07 | 系统维护弃牌堆；当主牌堆耗尽时，自动将弃牌堆洗牌并重置为新的主牌堆。 |
| FR-08 | 玩家在回合开始时从牌堆顶摸 2 张牌（若手牌为空，则摸 5 张）。 |
| FR-09 | 若摸牌时牌堆为空，系统自动将弃牌堆重新洗牌作为新牌堆后再摸牌。 |

### 2.3 回合流程

| 编号 | 描述 |
|------|------|
| FR-10 | 游戏按顺序（顺时针）轮换回合，每位玩家轮流行动。 |
| FR-11 | 每个回合内，当前玩家最多可出 3 张牌（任意组合：金钱牌、地产牌、行动牌）。 |
| FR-12 | 玩家可选择少出牌（0‑3 张），但不能超过 3 张。 |
| FR-13 | 回合结束时，若玩家手牌超过 7 张，必须弃牌至 7 张。 |
| FR-14 | 玩家弃牌时，弃牌进入弃牌堆。 |
| FR-15 | 当回合结束条件满足（出牌完毕或跳过）时，系统自动切换至下一位玩家的回合。 |

### 2.4 金钱牌（Money Cards）

| 编号 | 描述 |
|------|------|
| FR-16 | 玩家可将金钱牌从手牌打入自己的银行区，计入个人资产。 |
| FR-17 | 银行区的金钱牌一旦打出，不可撤回手牌。 |
| FR-18 | 系统实时显示每位玩家银行区的总金额。 |
| FR-19 | 金钱牌面值包括：1M、2M、3M、4M、5M、10M（M = 百万）。 |

### 2.5 地产牌（Property Cards）

| 编号 | 描述 |
|------|------|
| FR-20 | 玩家可将地产牌从手牌打入自己的地产区。 |
| FR-21 | 游戏包含 10 种颜色的地产套装，每种颜色有固定的套装张数（2 或 3 张）。 |
| FR-22 | 系统追踪每位玩家各颜色地产套装的收集情况，并判断是否形成完整套装。 |
| FR-23 | 系统支持万能地产牌（Wild Property Card），可归入玩家指定的颜色套装。 |
| FR-24 | 万能地产牌可在任意时候（回合内）移动至不同颜色套装，但不得在回合外移动。 |
| FR-25 | 完整套装不可被 Sly Deal / Forced Deal 拆分（仅 Deal Breaker 可整套偷取）。 |

### 2.6 行动牌（Action Cards）

#### 2.6.1 租金牌（Rent Cards）

| 编号 | 描述 |
|------|------|
| FR-26 | 玩家打出租金牌时，系统根据目标颜色套装的拥有张数计算对应租金金额。 |
| FR-27 | 颜色专属租金牌：仅对指定颜色的套装收取租金，向其他所有玩家收取。 |
| FR-28 | 万能租金牌（Any Color Rent）：向任意一名玩家收取任意一种颜色套装的租金。 |
| FR-29 | 租金收取时，目标玩家必须付款；若资产不足，须将全部资产转让（不会欠债）。 |
| FR-30 | 与"Double the Rent"联动：打出后下一张租金牌金额翻倍（须消耗一次出牌机会）。 |

#### 2.6.2 Pass Go

| 编号 | 描述 |
|------|------|
| FR-31 | 玩家打出 Pass Go 牌，立即从牌堆额外摸 2 张牌（加入手牌）。 |

#### 2.6.3 Debt Collector（催债人）

| 编号 | 描述 |
|------|------|
| FR-32 | 玩家打出 Debt Collector，指定任意一名对手，对方须向该玩家支付 5M。 |
| FR-33 | 若目标玩家资产不足 5M，须转让全部资产（不欠债）。 |

#### 2.6.4 Birthday（生日）

| 编号 | 描述 |
|------|------|
| FR-34 | 玩家打出 Birthday，其他所有玩家各向该玩家支付 2M。 |
| FR-35 | 若某玩家资产不足 2M，须转让全部资产（不欠债）。 |

#### 2.6.5 Sly Deal（狡猾交易）

| 编号 | 描述 |
|------|------|
| FR-36 | 玩家打出 Sly Deal，可从任意对手地产区偷取 1 张非完整套装的地产牌，移入自己地产区。 |
| FR-37 | Sly Deal 不可对完整颜色套装中的地产牌使用。 |

#### 2.6.6 Forced Deal（强制交换）

| 编号 | 描述 |
|------|------|
| FR-38 | 玩家打出 Forced Deal，与任意对手各自出 1 张非完整套装的地产牌互换。 |
| FR-39 | Forced Deal 不可对完整颜色套装中的地产牌使用。 |

#### 2.6.7 Deal Breaker（交易破坏者）

| 编号 | 描述 |
|------|------|
| FR-40 | 玩家打出 Deal Breaker，可整套偷取任意对手已集齐的完整颜色套装。 |

#### 2.6.8 House & Hotel（房屋 & 酒店）

| 编号 | 描述 |
|------|------|
| FR-41 | 玩家可在已完整的颜色套装上添加 1 张 House 牌，使该套装的租金增加对应金额。 |
| FR-42 | 玩家在已添加 House 的完整套装上可再添加 1 张 Hotel 牌，进一步提升租金。 |
| FR-43 | House 和 Hotel 牌在计算资产时具有对应面值（可用于付款）。 |
| FR-44 | 每个完整套装最多只能有 1 间 House 和 1 间 Hotel。 |

#### 2.6.9 Just Say No（拒绝）

| 编号 | 描述 |
|------|------|
| FR-45 | 任意玩家在任意时刻（包括非自己回合）可打出 Just Say No 牌以取消对自己生效的任意行动牌。 |
| FR-46 | Just Say No 可被另一张 Just Say No 反制（链式取消）。 |
| FR-47 | 打出 Just Say No 不消耗出牌次数。 |

#### 2.6.10 Double the Rent（租金翻倍）

| 编号 | 描述 |
|------|------|
| FR-48 | 玩家在同一回合内打出 Double the Rent 后，紧接着打出的租金牌金额翻倍；若再打一张 Double the Rent，则租金再次翻倍（最多 2 张叠加，即 4 倍）。 |
| FR-49 | Double the Rent 须与租金牌在同一回合内使用，否则无效；它本身消耗 1 次出牌机会。 |

### 2.7 付款机制

| 编号 | 描述 |
|------|------|
| FR-50 | 玩家支付欠款时，可用银行区金钱牌和/或地产区地产牌（含 House/Hotel）组合凑款。 |
| FR-51 | 付款方式由债务玩家选择（地产牌或金钱牌），债权人无权指定具体牌张。 |
| FR-52 | 支付的地产牌直接进入债权玩家的地产区；支付的金钱牌进入债权玩家的银行区。 |
| FR-53 | 若玩家资产不足以支付全额，转让全部资产即视为还清欠款，无需补偿差额。 |

### 2.8 胜利判定

| 编号 | 描述 |
|------|------|
| FR-54 | 当某位玩家在其回合中完成 3 套完整的颜色地产套装时，系统立即判定该玩家获胜。 |
| FR-55 | 胜利判定在每次地产牌入场（包括付款、偷取等方式到手）后实时检测。 |
| FR-56 | 系统在判定获胜后结束游戏，展示获胜玩家信息，并停止后续行动。 |

### 2.9 游戏界面与交互

| 编号 | 描述 |
|------|------|
| FR-57 | 系统展示所有玩家的地产区（牌的颜色、数量、是否完整套装）。 |
| FR-58 | 系统展示所有玩家的银行区总金额（金钱牌具体张数对己方可见，对其他玩家显示总额）。 |
| FR-59 | 系统展示当前玩家的手牌（仅当前玩家可见，其他玩家只见数量）。 |
| FR-60 | 系统显示当前回合已出牌张数及剩余可出牌张数（0‑3）。 |
| FR-61 | 系统在玩家出牌时提供合法性校验，防止违规操作（如超出 3 张、对完整套装使用 Sly Deal 等）。 |
| FR-62 | 系统提供"结束回合"操作，触发弃牌检查。 |
| FR-63 | 系统显示剩余牌堆张数。 |
| FR-64 | 系统记录并展示本局的出牌历史日志（行动记录）。 |

## 三、非功能需求（Non-functional Requirements）

### 3.1 玩家数量与可玩性

| 编号 | 描述 |
|------|------|
| NFR-01 | 游戏必须支持最少 2 名、最多 5 名玩家同时游玩。 |
| NFR-02 | 游戏须支持本地多人（同台）模式；若实现联网对战，网络延迟应不影响游戏流畅度。 |

### 3.2 规则一致性与逻辑正确性

| 编号 | 描述 |
|------|------|
| NFR-03 | 系统的所有游戏逻辑必须与《Monopoly Deal》官方规则完全一致，不得出现规则歧义或漏洞。 |
| NFR-04 | 规则引擎须覆盖所有边界情况（如资产不足、Just Say No 链式、万能牌归类等）。 |

### 3.3 性能

| 编号 | 描述 |
|------|------|
| NFR-05 | 任意操作（出牌、付款、胜负判定等）的响应时间应在 1 秒以内。 |
| NFR-06 | 系统在 2‑5 名玩家同时在线时不应出现性能瓶颈或崩溃。 |

### 3.4 可用性与用户体验

| 编号 | 描述 |
|------|------|
| NFR-07 | 界面布局清晰，玩家能够轻松区分手牌、银行区、地产区及弃牌堆。 |
| NFR-08 | 系统应为新手玩家提供规则提示或帮助文档（内嵌说明或工具提示）。 |
| NFR-09 | 非法操作时，系统应以明确的提示信息告知玩家原因，而非静默失败。 |
| NFR-10 | 游戏界面需区分各颜色地产套装，颜色编码须直观可辨。 |

### 3.5 可靠性与数据完整性

| 编号 | 描述 |
|------|------|
| NFR-11 | 系统须保证牌组在整局游戏中的数据完整性（不重复、不丢失）。 |
| NFR-12 | 每次状态变更（出牌、摸牌、付款）须原子性处理，避免出现中间态错误。 |

### 3.6 可维护性与可扩展性

| 编号 | 描述 |
|------|------|
| NFR-13 | 代码须遵循良好的面向对象设计原则（如单一职责、开闭原则），便于后续添加新牌或规则变体。 |
| NFR-14 | 游戏逻辑（规则引擎）须与界面层解耦，支持替换前端或添加 AI 对手模块。 |
| NFR-15 | 项目须提供充分的代码注释与文档，便于团队成员协同开发与交接。 |

### 3.7 安全性与公平性

| 编号 | 描述 |
|------|------|
| NFR-16 | 手牌信息对其他玩家不可见，系统须保证信息隐蔽性。 |
| NFR-17 | 洗牌算法须使用随机性足够的算法（如 Fisher-Yates），确保游戏公平。 |

### 3.8 可测试性

| 编号 | 描述 |
|------|------|
| NFR-18 | 核心游戏逻辑须可进行单元测试，覆盖所有牌的效果及边界情况。 |
| NFR-19 | 系统须提供调试或测试模式，允许指定初始手牌和牌堆顺序以复现特定场景。 |

## 五、关键约束汇总

1. 每回合最多出 **3** 张牌。
2. 回合结束手牌上限 **7** 张（超出须弃牌）。
3. 起始手牌 **5** 张；手牌为零时摸 **5** 张，否则每回合开始摸 **2** 张。
4. 欠款时**资产不足不欠债**，转让全部资产即可。
5. **完整套装**受到保护：不可被 Sly Deal / Forced Deal 拆单张（仅 Deal Breaker 可整套偷走）。
6. Just Say No 可**随时**打出，不消耗出牌次数，且可被另一张 Just Say No 反制。
7. 获胜需 **3 套完整颜色套装**（万能牌归入的颜色计入）。
